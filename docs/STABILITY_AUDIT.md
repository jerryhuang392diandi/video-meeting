# 视频会议稳定性审计与优化建议

## 结论

当前系统“不稳定”的主因不是单一 bug，而是 **前端房间状态机、WebRTC 信令、屏幕共享链路、虚拟背景 CPU 占用、以及服务端房间内存态之间耦合过深**。现状可以跑起来，但在以下场景里非常容易出现竞争条件：

- 多人几乎同时加入会议。
- 用户刷新页面、断网重连、同账号多标签页重入。
- 摄像头/麦克风/屏幕共享在通话过程中频繁开关。
- 屏幕共享与虚拟背景、录屏同时启用。
- 参与者数量增长后，前端需要维护的 `RTCPeerConnection` 数量成倍增加。

## 这次已落地的修复

### 1. WebRTC 信令碰撞保护

前端此前采用“谁加入、谁就立刻 `createOffer`”的方式，并且在补加 track 时也会直接重新发 offer。多人同时加入或同时切换媒体时，很容易发生 **offer glare**：

- 双方都处于 `have-local-offer`。
- 新 offer 到来时直接 `setRemoteDescription()` 报错。
- ICE candidate 先于 remote description 到达时直接 `addIceCandidate()` 失败。

这次在前端补了三层保护：

- `makingOffer` 状态，避免重复并发发起 offer。
- 基于 socket id 的 polite/impolite 冲突消解与 `rollback`。
- `pendingCandidates` 队列，在 remote description 就绪前先缓存 candidate。

这会明显改善：

- 多人同时加入时的黑屏/偶发无流。
- 摄像头开启关闭后的“有名字没画面”。
- 屏幕共享切换回来后个别人收不到新视频轨的问题。

### 2. ICE 失败后的恢复尝试

原实现里连接失败会直接移除远端卡片并删掉连接，恢复能力很弱。现在增加了：

- `iceConnectionState === 'failed'` 时尝试 `restartIce()`。
- 重启 ICE 后再触发一次 offer 流程。

这不等于彻底解决网络问题，但能减少短时网络抖动后“需要手动重进会议”的概率。

## 我看到的核心结构问题

### A. 这是典型的 Mesh 架构，人数一多天然不稳

当前前端对每个远端用户都建立一个 `RTCPeerConnection`，属于 **全互联 Mesh**。这意味着：

- 每个参会者上行带宽压力随人数线性增长。
- 整体连接数随人数接近平方增长。
- 屏幕共享时，分享者需要把高码率画面分别推给所有人。

所以“随着人员加入离开越来越不稳、共享屏幕更卡”，本质上很符合 Mesh 的性能边界。

**建议优先级最高的长期方案：迁移到 SFU。**

可选方向：

- LiveKit
- mediasoup
- Janus
- Jitsi Meet（如果接受集成式方案）

如果目标是 6 人以上稳定开会、并且要屏幕共享清晰，继续在 Mesh 上打补丁，收益会越来越低。

### B. 服务端房间状态完全在内存里，恢复能力弱

`rooms`、`sid_to_user`、`user_active_sids` 都在单进程内存中维护。问题包括：

- 进程重启后实时状态全部丢失。
- 多实例部署几乎不可行。
- Socket 事件与数据库状态之间没有严格事务边界。
- 房主离开、共享者切换、同账号挤下线等逻辑高度依赖“事件顺序刚好正确”。

**建议：**

- 短期：把房间在线态迁到 Redis。
- 中期：给会议成员、媒体状态、共享状态定义明确状态机。
- 长期：信令服务独立化，应用层只做鉴权和业务数据。

### C. 屏幕共享事件和真实媒体轨状态没有完全解耦

现在 `screen_share_started/stopped` 既在表达 UI 模式，又隐含表达“当前主视频轨是谁的屏幕”。这会导致：

- UI 已经切到共享模式，但某些 peer 的 `replaceTrack()` 还没真正同步完成。
- 某用户断线后，服务端会把共享状态“继承”给同账号其他 sid，但前端未必真的还持有有效屏幕轨。
- 共享结束后还要走虚拟背景/摄像头恢复逻辑，状态切换过多。

**建议：**

把“共享状态”拆成三层：

1. `intent`: 用户请求开始/结束共享。
2. `transport`: 当前 peer 是否已经收到新的屏幕 track。
3. `ui`: 前端是否进入共享布局。

只有在多数连接完成媒体切换后，再统一广播 UI 进入共享模式，会更稳定。

### D. 虚拟背景是纯前端逐帧分割，CPU 压力非常大

当前虚拟背景链路：

- `requestAnimationFrame` 驱动。
- MediaPipe SelfieSegmentation 前端实时推理。
- canvas 重绘后 `captureStream(15fps)` 再送进 WebRTC。

这会带来：

- CPU 飙升。
- 笔记本风扇狂转。
- 弱设备丢帧、音视频不同步。
- 与屏幕共享/录屏同时工作时更容易卡顿。

**建议：**

- 默认关闭虚拟背景，不作为默认能力。
- 只在 1v1 或 1v2 时允许开启。
- 动态降级：检测卡顿后降到 8fps / 更低分辨率。
- 增加“模糊背景”替代“图片背景”，成本更低。

### E. 缺少统一的连接诊断面板

目前有 `debugRoom=1` 日志，但没有结构化的观测指标。排查不稳定时，缺少这些核心信息：

- 每个 peer 的 `connectionState` / `iceConnectionState`。
- 当前发送/接收码率。
- 丢包率、抖动、RTT。
- 当前视频轨分辨率、帧率。
- 屏幕共享 track 是否存活。

**建议尽快补一个调试面板：**

- 前端每 3~5 秒抓一次 `pc.getStats()`。
- UI 上展示发送/接收 bitrate、fps、packet loss、RTT。
- 出问题时可一键导出 JSON。

## 建议的优化优先级

### P0：先止血（1~3 天）

1. 合并并梳理前端会议状态机：
   - `idle`
   - `joining`
   - `joined`
   - `sharing_screen`
   - `recovering_media`
   - `leaving`
2. 给所有媒体切换流程加串行锁，避免：
   - 正在共享时又切摄像头。
   - 虚拟背景恢复与屏幕共享结束并发。
   - 多次点击按钮触发重入。
3. 对 `getStats()` 做最小监控。
4. 屏幕共享默认降到 720p / 8~10fps，并按网络状况动态调节。

### P1：稳定加入/离开/重连（3~7 天）

1. 明确“同账号多标签页”策略：
   - 是允许多端在线，还是强制单端？
   - 如果强制单端，前端要识别“被替换登录”并优雅收尾。
2. 为重连建立 session 恢复机制：
   - 断线重连后优先复用身份。
   - 重新拉取 participant snapshot。
   - 对缺失 peer 定向补建连接。
3. 参与者离开时，不要立刻销毁 UI，可先进入短暂 `reconnecting` 状态。

### P2：架构升级（1~3 周）

1. 用 Redis 托管房间在线态。
2. 将媒体信令抽离成独立模块。
3. 评估并迁移到 SFU。

## 推荐的下一步实施顺序

1. 先上线这次信令修复并观察真实会议情况。
2. 紧接着补 `pc.getStats()` 调试面板。
3. 给媒体切换加互斥锁与防抖。
4. 再决定是否继续修 Mesh，还是直接转 SFU。

## 如果让我继续接着做

我建议下一轮直接做这三件事：

1. **媒体状态机重构**：把摄像头、屏幕共享、虚拟背景的切换流程串行化。
2. **连接诊断面板**：实时展示每个 peer 的码率/丢包/RTT/fps。
3. **屏幕共享专项优化**：降低默认帧率、补共享恢复逻辑、减少 UI 与媒体状态耦合。
