# 单机公网安全上线顺序

这份文档面向当前项目的现实情况：

- 你只有一台服务器；
- 预算有限，暂时不买第二台入口机；
- 项目是 `Flask + Flask-SocketIO + LiveKit`；
- 你希望先把服务放到公网，但尽量降低被扫、被爆破、被滥用的风险。

这不是“绝对安全”方案，而是当前条件下更稳、更现实的一套上线顺序。

---

## 1. 先搞清楚 `.env` 在哪里改

当前项目有两个常见环境：

### 本地开发

本地直接运行 `python app.py` 时，代码会读取**项目根目录**的 `.env`：

- 代码位置：[app.py](D:/study/大三下/视觉媒体通信/final_project/video_meeting_replace/app.py:57)
- 读取方式：`load_dotenv(os.path.join(BASE_DIR, ".env"))`

也就是说，本地开发时你应该改：

```text
D:\study\大三下\视觉媒体通信\final_project\video_meeting_replace\.env
```

如果这个文件还不存在，就在项目根目录新建一个 `.env`。

### 线上服务器

如果你按仓库里的部署文档用 `systemd + gunicorn` 部署，推荐改的是：

```text
/opt/video-meeting/.env
```

原因是部署文档里的 systemd 服务会显式加载这个文件：

- 参考：[docs/DEPLOYMENT_GUIDE.md](D:/study/大三下/视觉媒体通信/final_project/video_meeting_replace/docs/DEPLOYMENT_GUIDE.md:327)

也就是说：

- 本地改：项目根目录 `.env`
- 线上改：`/opt/video-meeting/.env`

### 改完 `.env` 以后要做什么

本地：

```bash
python app.py
```

或者先停掉再重启。

线上：

```bash
sudo systemctl restart video-meeting
sudo systemctl status video-meeting
```

只改 `.env` 不重启服务，通常不会生效。

## 2. 你的上线目标应该是什么

你现在不是要做“高并发商用平台”，而是要先做到：

1. 正常用户能打开首页、登录、进房间。
2. 常见扫描器、爆破器、低质量攻击流量不容易直接把服务打穿。
3. 管理员账号和源站不要因为默认配置而裸奔。
4. 即使被打，也尽量先卡在 Nginx、限流、人机验证、fail2ban 这一层，而不是直接把 Flask 顶死。

所以顺序上应该是：

1. 先收紧应用配置
2. 再收紧 Nginx
3. 再加 fail2ban
4. 最后再考虑外部 IP 黑名单

不要反过来先堆一堆黑名单。

---

## 3. 上线前先准备这些信息

在你动服务器前，先准备好：

- 你的公网域名，例如 `meeting.example.com`
- 你的服务器公网 IP
- LiveKit Cloud 的 `LIVEKIT_URL`、`LIVEKIT_API_KEY`、`LIVEKIT_API_SECRET`
- 一个新的管理员用户名
- 一个新的强管理员密码
- 一个新的强 `SECRET_KEY`
- 你的固定 SSH 登录 IP
- Cloudflare Turnstile 的 `site key` 和 `secret key`

生成强 `SECRET_KEY` 的方式：

```bash
python3 -c "import secrets; print(secrets.token_urlsafe(48))"
```

生成强密码的最低原则：

- 至少 16 位
- 不要用生日、学号、手机号、项目名
- 不要继续用 `root1234`、`admin123`、`123456`

## 4. 第零步：先做现状检查，不要直接改

如果你现在服务器已经“改乱了”，先不要急着继续改 `.env` 或 Nginx。先把当前到底是谁在生效查清楚。

按这个顺序执行：

```bash
whoami
hostname
pwd

sudo systemctl status video-meeting --no-pager -l
sudo systemctl cat video-meeting
sudo journalctl -u video-meeting -n 120 --no-pager

ps -ef | grep -E "gunicorn|python.*app.py" | grep -v grep
sudo ss -lntp | grep -E ':80|:443|:5000|:8000'

sudo nginx -t
sudo systemctl status nginx --no-pager -l
sudo tail -n 80 /var/log/nginx/error.log
sudo tail -n 80 /var/log/nginx/access.log

sudo sed -n '1,120p' /opt/video-meeting/.env
curl -I http://127.0.0.1:8000
curl -I http://127.0.0.1:5000
```

你要重点识别这几件事：

- `systemctl cat video-meeting` 里是否有 `EnvironmentFile=/opt/video-meeting/.env`
- `ExecStart` 到底是 `gunicorn --worker-class eventlet --workers 1 --bind 127.0.0.1:8000 app:app`，还是 `python app.py`
- `ss -lntp` 里到底是谁在监听 `5000` 或 `8000`
- Nginx 是不是在对外监听 `80/443`
- 扫描流量是不是已经直接打到应用层

### 4.1 如何判断当前运行方式是不是错的

如果你在线上看到下面这些现象，说明当前不是推荐部署方式：

- `systemctl status video-meeting` 里出现 `Werkzeug appears to be used in a production deployment`
- 日志里出现 `This is a development server`
- `ExecStart` 是 `/opt/video-meeting/venv/bin/python /opt/video-meeting/app.py`
- 应用直接监听 `0.0.0.0:5000` 或公网 IP `:5000`

这说明你当前跑的是 Flask 开发服务器，不是生产部署。

### 4.2 线上正确目标状态

线上正确目标应该是：

- Nginx 监听公网 `80/443`
- Gunicorn 只监听 `127.0.0.1:8000`
- systemd 使用 `EnvironmentFile=/opt/video-meeting/.env`
- systemd 用 `gunicorn + eventlet` 启动，不是 `python app.py`

如果你是“全部重来一遍”，就按下面的 `.env + systemd + Nginx` 结构重建，不要继续在旧结构上打补丁。

## 5. 第一步：先改线上 `.env`

如果你现在最不确定的是“到底该改哪个文件”，那就先按下面顺序确认。

### 5.1 先确认线上 `.env` 位置

标准部署方式下，线上环境变量文件通常就是：

```text
/opt/video-meeting/.env
```

先执行：

```bash
cd /opt/video-meeting
pwd
ls -la
sudo systemctl cat video-meeting
```

重点看 systemd 服务文件里有没有这一行：

```ini
EnvironmentFile=/opt/video-meeting/.env
```

如果有，就说明服务启动时读取的就是这个文件。

### 5.1.1 如果你看到的不是 `EnvironmentFile`，而是很多 `Environment=...`

有些旧部署不是从 `.env` 读取，而是直接把变量写在：

- `/etc/systemd/system/video-meeting.service`
- 或 `/etc/systemd/system/video-meeting.service.d/override.conf`

典型特征就是你执行：

```bash
sudo systemctl cat video-meeting
```

看到的是这种内容：

```ini
[Service]
Environment=PUBLIC_HOST=example.com
Environment=PUBLIC_SCHEME=https
Environment=ADMIN_USERNAME=root
Environment=ADMIN_PASSWORD=...
Environment=LIVEKIT_URL=wss://...
```

这就说明你当前不是“改 `.env` 就生效”的部署，而是“改 systemd 里的 `Environment=` 才生效”。

这时候你有两个选择：

1. 继续沿用当前方式，直接修改 `.service` / `override.conf`
2. 迁移到更好维护的 `.env + EnvironmentFile=...` 方式

推荐第二种，因为后面加配置会更清楚，也更容易备份。

### 5.2 先备份旧文件

```bash
sudo cp /opt/video-meeting/.env /opt/video-meeting/.env.bak.$(date +%F-%H%M%S)
```

如果提示原文件不存在，说明你这次是在新建 `.env`，可以继续。

如果你当前还是 `Environment=...` 方式，再额外备份一下服务文件：

```bash
sudo cp /etc/systemd/system/video-meeting.service /etc/systemd/system/video-meeting.service.bak.$(date +%F-%H%M%S)
sudo mkdir -p /etc/systemd/system/video-meeting.service.d
sudo cp /etc/systemd/system/video-meeting.service.d/override.conf /etc/systemd/system/video-meeting.service.d/override.conf.bak.$(date +%F-%H%M%S) 2>/dev/null || true
```

### 5.3 用 EOF 方式重写 `.env`

如果你不想用 `nano`，就直接用 EOF。把下面整段复制到服务器里，再把占位值替换成你自己的真实值：

```bash
sudo tee /opt/video-meeting/.env > /dev/null <<'EOF'
SECRET_KEY=replace-with-a-long-random-secret
DATABASE_URL=sqlite:////opt/video-meeting/instance/app.db

PUBLIC_SCHEME=https
PUBLIC_HOST=meeting.example.com

LIVEKIT_URL=wss://your-project.livekit.cloud
LIVEKIT_API_KEY=replace-with-livekit-api-key
LIVEKIT_API_SECRET=replace-with-livekit-api-secret

ADMIN_USERNAME=meetingadmin
ADMIN_PASSWORD=replace-with-strong-admin-password

PUBLIC_REGISTRATION_ENABLED=0
STRICT_SECURITY_CHECKS=1

SESSION_COOKIE_SECURE=1
REMEMBER_COOKIE_SECURE=1
SESSION_COOKIE_SAMESITE=Lax
REMEMBER_COOKIE_SAMESITE=Lax

LOGIN_RATE_LIMIT_PER_IP=20
LOGIN_RATE_LIMIT_PER_USER=8
LOGIN_RATE_LIMIT_WINDOW_SECONDS=600
REGISTER_RATE_LIMIT_PER_IP=5
REGISTER_RATE_LIMIT_WINDOW_SECONDS=3600
PASSWORD_RESET_RATE_LIMIT_PER_IP=5
PASSWORD_RESET_RATE_LIMIT_WINDOW_SECONDS=3600

TURNSTILE_SITE_KEY=
TURNSTILE_SECRET_KEY=
EOF
```

然后执行：

```bash
sudo chmod 600 /opt/video-meeting/.env
```

### 5.4 EOF 是什么意思

- `<<'EOF'` 表示从这里开始读正文
- 最后那个单独一行的 `EOF` 表示正文结束
- 中间所有内容都会被写入 `/opt/video-meeting/.env`

你可以把它理解成“整段复制粘贴生成文件”。

### 5.5 哪些值必须换成你自己的

- `SECRET_KEY`
- `PUBLIC_HOST`
- `LIVEKIT_URL`
- `LIVEKIT_API_KEY`
- `LIVEKIT_API_SECRET`
- `ADMIN_USERNAME`
- `ADMIN_PASSWORD`
- `TURNSTILE_SITE_KEY`
- `TURNSTILE_SECRET_KEY`

其中最容易踩坑的是：

- `ADMIN_USERNAME` 不要再用 `root`
- `LIVEKIT_*` 三项必须来自同一个 LiveKit 项目
- `TURNSTILE_SITE_KEY` 和 `TURNSTILE_SECRET_KEY` 必须来自同一个 Turnstile 站点配置

如果你用的是自建 LiveKit，再额外记住一条：

- `LIVEKIT_URL` 是你自己给 LiveKit 准备的公网 `wss://` 地址
- `LIVEKIT_API_KEY` 和 `LIVEKIT_API_SECRET` 要去自建 LiveKit 的服务端配置里找
- 它们不是 Cloudflare 的 key，也不是 gunicorn 或 Flask 自动生成的

不要把下面三类配置混在一起：

- `LIVEKIT_API_KEY` / `LIVEKIT_API_SECRET`：LiveKit 鉴权
- `TURNSTILE_SITE_KEY` / `TURNSTILE_SECRET_KEY`：Cloudflare Turnstile 验证码
- `SECRET_KEY`：Flask 自己的会话签名

### 5.6 Turnstile 的 `secret key` 不是随便填

这点必须说清楚：

- `SECRET_KEY` 可以自己生成
- `TURNSTILE_SECRET_KEY` 不可以自己随便写一个随机字符串

`TURNSTILE_SECRET_KEY` 必须去 Cloudflare Turnstile 后台创建站点后复制。服务端会拿它去请求 Cloudflare 校验验证码；如果这把 key 不是官方给你的那把，验证会直接失败。

### 5.7 如果你暂时不想开人机验证

那就把这两个都留空：

```env
TURNSTILE_SITE_KEY=
TURNSTILE_SECRET_KEY=
```

这里最关键的是：

- `ADMIN_USERNAME` 不要再用 `root`
- `PUBLIC_REGISTRATION_ENABLED=0`
- `STRICT_SECURITY_CHECKS=1`
- `SESSION_COOKIE_SECURE=1`
- Turnstile 只有在 `TURNSTILE_SITE_KEY` 和 `TURNSTILE_SECRET_KEY` 都填了时才会启用

### 5.8 写完后先检查文件内容

```bash
sudo sed -n '1,80p' /opt/video-meeting/.env
grep -E 'PUBLIC_HOST|PUBLIC_SCHEME|LIVEKIT_URL|ADMIN_USERNAME|TURNSTILE_SITE_KEY' /opt/video-meeting/.env
```

### 5.9 如果你现在就是 `Environment=` 方式，应该怎么改

你这类情况最稳的做法不是继续把更多密钥塞进 `.service`，而是迁移成：

- `/opt/video-meeting/.env` 存配置
- `/etc/systemd/system/video-meeting.service` 只保留 `EnvironmentFile=/opt/video-meeting/.env`

#### 第一步：先写 `.env`


```bash

```

```bash

sudo systemctl status video-meeting

```

此时你应该看到：

```ini

```

而不是一长串散落的 `Environment=...` 密钥。

### 5.10 为什么线上一定不要用 `python app.py`

`python app.py` 只适合本地开发，不适合作为公网服务入口。原因不是“好不好看”，而是：

- 它会启动 Werkzeug 开发服务器
- 日志里会明确提示 `This is a development server`
- Flask-SocketIO 的长连接和稳定性不应该依赖这个运行方式
- 它很容易直接监听 `0.0.0.0:5000` 或公网 IP，暴露不必要的入口

如果你在线上运行 `sudo systemctl status video-meeting`，看到：

- `Werkzeug appears to be used in a production deployment`
- `Running on http://127.0.0.1:5000`
- `Running on http://你的公网IP:5000`

那就不要继续在这个基础上做“微调”，而是应该直接切回上面的 `gunicorn + eventlet + 127.0.0.1:8000` 结构。

### 5.11 你这张图里已经暴露了什么风险

如果这张图发到公开聊天、公开仓库、群聊或答辩材料里，建议你立刻更换：

- `SECRET_KEY`
- `ADMIN_PASSWORD`
- `TURNSTILE_SECRET_KEY`
- `LIVEKIT_API_SECRET`
- 如果同一张图里还有 `LIVEKIT_API_KEY`，也一起换

原因很简单：

- `ADMIN_PASSWORD` 已经明文暴露
- `LIVEKIT_API_SECRET` 已经明文暴露

如果只是你本机截图、从未发给别人，风险相对小一些；但只要离开你自己本地设备，就按“已泄露”处理更稳。

## 7. 第三步：确认 Flask 不是直接裸露在公网

你的目标应该是：

- 浏览器访问 Nginx：`80/443`
- Nginx 再转发到本机 `127.0.0.1:8000`
- Flask/Gunicorn 不直接监听公网

查看服务文件：

```bash
sudo systemctl cat video-meeting
```

如果需要修改：

```bash

```

改完后：

```bash
sudo systemctl daemon-reload
```

## 8. 第四步：给 Nginx 加限流

这一步是最值钱的，因为很多垃圾流量应该在 Nginx 层就被挡掉。

### 先写限流公共配置

```bash
sudo tee /etc/nginx/conf.d/video_meeting_rate_limit.conf > /dev/null <<'EOF'
limit_req_zone $binary_remote_addr zone=auth_limit:10m rate=10r/m;
limit_req_zone $binary_remote_addr zone=general_limit:10m rate=30r/m;
limit_conn_zone $binary_remote_addr zone=addr_conn:10m;
EOF
```

### 再编辑你的站点配置

```bash
sudo nano /etc/nginx/sites-available/video-meeting
```

在原有 `server` 里，把认证相关路径单独写出来。核心思路是：

- `/login`
- `/register`
- `/forgot-password`

单独限制比普通页面更严格。

示例：

```nginx
server {
    listen 80;
    server_name 你的域名;

    client_max_body_size 150m;

    location = /login {
        limit_req zone=auth_limit burst=10 nodelay;
        limit_conn addr_conn 20;
        proxy_pass http://127.0.0.1:8000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location = /register {
        limit_req zone=auth_limit burst=5 nodelay;
        limit_conn addr_conn 20;
        proxy_pass http://127.0.0.1:8000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location = /forgot-password {
        limit_req zone=auth_limit burst=5 nodelay;
        limit_conn addr_conn 20;
        proxy_pass http://127.0.0.1:8000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location / {
        limit_req zone=general_limit burst=60 nodelay;
        limit_conn addr_conn 40;
        proxy_pass http://127.0.0.1:8000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection $connection_upgrade;
        proxy_read_timeout 3600;
        proxy_send_timeout 3600;
    }
}
```

### 改完后检查并重载

```bash
sudo nginx -t
sudo systemctl reload nginx
```

### 这一步成功的判断标准

- `sudo nginx -t` 返回成功
- 网站仍能打开
- 正常登录不受影响
- 连续快速狂刷登录页时开始出现限流

## 9. 第五步：启用 fail2ban

它适合你这种单机场景，成本低，收益高。

### 安装

```bash
sudo apt update
sudo apt install -y fail2ban
```

### 先启用 Nginx 常见规则

创建：

```bash
sudo tee /etc/fail2ban/jail.d/nginx-common.local > /dev/null <<'EOF'
[nginx-http-auth]
enabled = true

[nginx-badbots]
enabled = true
EOF
```

### 再给你的登录接口单独加一个规则

创建过滤器：

```bash
sudo tee /etc/fail2ban/filter.d/nginx-video-meeting-auth.conf > /dev/null <<'EOF'
[Definition]
failregex = ^<HOST> - .* "(GET|POST) /login HTTP/.*" (403|429) .*
ignoreregex =
EOF
```

创建 jail：

```bash
sudo tee /etc/fail2ban/jail.d/video-meeting-auth.local > /dev/null <<'EOF'
[video-meeting-auth]
enabled = true
port = http,https
filter = nginx-video-meeting-auth
logpath = /var/log/nginx/access.log
findtime = 600
maxretry = 12
bantime = 3600
EOF
```

### 启动并检查

```bash
sudo systemctl restart fail2ban
sudo systemctl status fail2ban
sudo fail2ban-client status
sudo fail2ban-client status video-meeting-auth
```

### 这一步成功的判断标准

- `fail2ban` 服务是 active
- `video-meeting-auth` jail 存在
- 后续如果有爆破，会出现封禁记录

### 查看封禁列表

```bash
sudo fail2ban-client status video-meeting-auth
```

### 手动解封一个 IP

```bash
sudo fail2ban-client set video-meeting-auth unbanip 1.2.3.4
```

## 10. 第六步：收紧服务器防火墙

如果你没有特殊需求，公网只应该开放：

- `80/tcp`
- `443/tcp`
- `22/tcp` 但最好只允许你自己的 IP

### 使用 UFW

```bash
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw allow from 你的固定IP to any port 22 proto tcp
sudo ufw enable
sudo ufw status verbose
```

### 如果你的 SSH IP 不固定

那你可以先临时保留：

```bash
sudo ufw allow 22/tcp
```

但后面还是建议收紧。

### 这一步成功的判断标准

- 网站正常可访问
- 你自己还能 SSH 登录
- 非必须端口没有继续暴露

执行检查：

```bash
sudo ss -lntp
```

看哪些端口真的在监听。

## 11. 第七步：管理员后台额外保护

你现在至少应该做到：

- 管理员账号不叫 `root`
- 管理员密码足够强
- 管理员后台不要共享给别人

如果你有固定 IP，最稳的是直接在 Nginx 里给 `/admin` 加 IP 白名单：

```nginx
location /admin {
    allow 你的固定IP;
    deny all;

    proxy_pass http://127.0.0.1:8000;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

如果你 IP 不固定，那就先不要急着上白名单，但至少保证：

- 关闭公开注册
- 开启 Turnstile
- 开启 fail2ban

## 12. 第八步：是否要上 IP 黑名单

我的建议是：

- 不要一开始就上大规模 blocklist
- 先跑一段时间观察日志
- 只有在你确实遇到大量扫描器时，再加

### 更推荐的优先级

1. `fail2ban`
2. `CrowdSec`
3. 少量高质量 blocklist
4. 最后才是大而杂的全量黑名单

### 原因

静态 IP 黑名单的常见问题：

- 误封正常用户
- 封掉学校、运营商、共享出口 IP
- 封掉移动网络大出口
- 维护成本高

对你这种会议类项目来说，误封比“少挡住几个扫描器”更麻烦。

## 13. 第九步：上线后的第一轮手工验证

不要改完就结束。至少做下面这些检查。

### 基础访问检查

1. 首页能打开
2. 登录页能打开
3. 帮助页能打开
4. `/room/<room_id>` 在 LiveKit 正常配置时可进入

4. 找回密码页仍可提交，但会受限流和 Turnstile 约束

### Turnstile 检查

1. 登录页能显示人机验证
2. 不做验证直接提交，应该失败
3. 做完验证再提交，应该正常
4. 用中国大陆网络再测一次

### 房间功能检查

1. 两个客户端能加入同一房间
2. 不刷新即可互相看到
3. 聊天上传可用
4. `/admin` 仍可正常打开

## 15. 遇到问题时先排查什么

### 改了 `.env` 但没生效

先做：

```bash
sudo systemctl restart video-meeting
sudo systemctl status video-meeting
```

再看：

```bash
sudo journalctl -u video-meeting -n 100 --no-pager
```

### 打开网站是 502

先看 Flask/Gunicorn 是否挂了：

```bash
sudo systemctl status video-meeting
```

再看 Nginx：

```bash
sudo nginx -t
sudo tail -n 100 /var/log/nginx/error.log
```

### Turnstile 一直失败

优先排查：

1. `TURNSTILE_SITE_KEY` 和 `TURNSTILE_SECRET_KEY` 是否填对
2. 服务器能否访问 `challenges.cloudflare.com`
3. 中国大陆网络是否能正常加载前端脚本

如果主要是大陆用户打不开，就先关闭 Turnstile，再依赖 Nginx + fail2ban + 应用限流。

### 服务启动直接失败

如果你开了：

```env
STRICT_SECURITY_CHECKS=1
```

那就检查：

- `SECRET_KEY` 是否过短或还是示例值
- `ADMIN_PASSWORD` 是否过弱
- `ADMIN_USERNAME` 是否还是 `root`

## 16. 你当前最推荐的最终状态

如果你现在就要上线，我建议最终至少做到这个配置：

### 应用层

- `PUBLIC_REGISTRATION_ENABLED=0`
- `STRICT_SECURITY_CHECKS=1`
- 强 `SECRET_KEY`
- 强管理员用户名和密码
- 登录/注册/找回密码限流已开启

### 代理层

- Nginx 只对外开放 `80/443`
- Gunicorn 只监听 `127.0.0.1:8000`
- 对 `/login`、`/register`、`/forgot-password` 做单独限流

### 主机层

- UFW 已开启
- 只开放必要端口
- fail2ban 已开启

### 可选层

- Turnstile 已实测可用才开启
- IP 黑名单只做辅助，不做主防线
