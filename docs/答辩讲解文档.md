# 视频会议项目答辩讲解文档

这份讲解稿按“先讲清系统是什么，再讲代码怎么分工，最后解释为什么这样选技术”的顺序整理，适合答辩时直接展开。

## 1. 项目一句话介绍

这个项目是一个基于 `Flask + Flask-SocketIO + LiveKit` 的在线视频会议系统。用户可以注册登录、创建或加入会议，在会议里进行音视频通话、屏幕共享、聊天、发送附件，主持人还可以结束会议和管理房间。

如果老师先问“你这个项目核心解决了什么问题”，可以直接回答：

> 它把账号体系、房间管理、实时状态同步和多人音视频会议整合成了一个完整可运行的 Web 系统。

## 2. 技术栈为什么这样分

### Flask 负责什么

Flask 是后端业务框架，负责：

- 登录、注册、退出
- 创建会议、加入会议、历史会议
- 管理员后台
- 聊天和附件接口
- 生成 LiveKit token

可以把 Flask 理解成“系统的大脑和业务入口”。

### Socket.IO 负责什么

Socket.IO 负责实时状态同步，适合处理：

- 有人加入或离开会议
- 聊天消息广播
- 主持人结束会议
- 屏幕共享相关 UI 状态通知

它传的是“状态消息”，不是音视频本体。

### LiveKit 负责什么

LiveKit 负责真正的音视频媒体传输，包括：

- 发布本地摄像头和麦克风
- 订阅远端用户轨道
- 屏幕共享轨道发布
- 多人会议下的媒体转发

它本质上是 SFU 架构，而不是浏览器之间互相硬连的 Mesh。

## 3. 为什么项目里要用 SFU

答辩时这部分很关键，因为它能体现你对系统规模的理解。

可以这样讲：

> 如果采用 Mesh，也就是每个用户都直接和其他每个用户连，那么人数一多，连接数量会迅速增加，带宽、CPU 和维护复杂度都会上升。  
> 这个项目选择 LiveKit 的 SFU 方案，是因为每个用户只需要把自己的媒体发到服务器一次，再由服务器转发给其他人，更适合多人会议。

所以这个项目的技术路线不是“自己硬写完整 WebRTC 信令栈”，而是：

- 业务逻辑自己做
- 媒体分发交给更稳定的 LiveKit

## 4. 代码结构怎么讲

### `app.py`

这是后端核心文件，里面集中放了：

- Flask 初始化
- SQLAlchemy 模型
- 页面路由
- REST API
- Socket.IO 事件
- 运行时房间状态

答辩时可以说：

> 这个项目后端目前是单体文件组织，优点是开发阶段定位快，缺点是后续功能继续增加时需要进一步按模块拆分。

### `templates/`

这里放页面模板，例如：

- 登录页
- 注册页
- 首页
- 会议室页面
- 管理后台

其中房间页拆成 `_room_layout.html` 和 `_room_scripts.html`，这样结构和脚本职责更清楚。

### `static/`

这里放前端静态资源，主要分工是：

- `room_livekit.js`：管理 LiveKit 房间连接和媒体轨道
- `room_ui.js`：房间布局、卡片排序、焦点切换
- `room_chat.js`：聊天消息和附件渲染
- `room_diagnostics.js`：RTC 诊断信息展示
- `room_utils.js`：通用工具函数

### `translations.py`

这个文件负责中英文翻译表，模板和后端通过 `t(...)` 读取文案。

## 5. 用户使用流程怎么讲

### 第一步：登录

用户先在登录页输入用户名和密码。后端验证成功后写入会话，并记录 `session_version`，这样可以控制同账号会话和管理员踢人逻辑。

### 第二步：创建或加入会议

登录后，用户可以：

- 创建会议，系统生成 6 位房间号和房间密码
- 或输入房间号和密码加入已有会议

数据库会记录会议信息，运行时内存还会记录当前房间在线态。

### 第三步：进入会议室

进入房间页后会做两件事：

1. 前端通过 Socket.IO 向后端声明“我进入房间了”
2. 前端再向后端申请 LiveKit token，随后连接 LiveKit 房间

这一步体现了系统的分层：

- Socket.IO 负责房间成员状态
- LiveKit 负责音视频媒体

### 第四步：开始会议交互

进房后用户可以：

- 开关麦克风
- 开关摄像头
- 屏幕共享
- 发送聊天消息
- 上传图片、视频、文档附件
- 使用虚拟背景
- 本地录屏

主持人还能：

- 结束会议
- 清空聊天
- 通过后台管理用户和会议

## 6. 这个项目有哪些实现亮点

### 亮点 1：业务层和媒体层是分开的

很多课程项目会把“实时消息”和“音视频传输”混在一起写，后面会越来越乱。这个项目已经把职责分开：

- Flask 处理业务
- Socket.IO 处理状态同步
- LiveKit 处理媒体

这比把所有问题都堆进一套前端 WebRTC 逻辑更容易维护。

### 亮点 2：不仅有通话，还有完整会议功能

它不只是“能看到视频”，而是包含：

- 账号体系
- 房间管理
- 会议历史
- 附件与聊天
- 主持人权限
- 管理后台

所以它更接近一个完整系统，而不只是单点 demo。

### 亮点 3：考虑了诊断和运维

项目里已经有基础的 RTC 诊断面板，还支持录屏转 MP4、管理员统计和重置密码处理。这说明设计目标不只是“写出功能”，而是“让它更接近可维护系统”。

## 7. 目前还有哪些局限

这部分不要回避，直接讲会更专业。

### 局限 1：运行时在线态还是单进程内存

现在房间在线用户、Socket 映射等状态主要放在进程内存里。优点是简单，缺点是：

- 服务重启会丢失在线态
- 多实例部署不方便
- 后续更适合迁移到 Redis

### 局限 2：`app.py` 仍然偏大

当前阶段便于开发和排查，但长期来看，后端最好继续拆成认证、房间、聊天、后台等模块。

### 局限 3：高负载功能对资源要求高

例如：

- 屏幕共享
- 虚拟背景
- 录屏转 MP4

这些功能会额外占用浏览器端或服务端资源，不可能“零成本”存在。

## 8. 如果老师问：为什么不用 C 语言

这是你这次特别需要准备的回答。

### 先给结论

可以用 C 语言做一部分，但**不适合把当前这个项目整体用 C 语言重写**。

### 为什么“理论上可以”

C 语言当然能写网络程序、音视频程序、服务器程序。像很多高性能网络库、媒体编解码库，本身就是 C 或 C++ 写的。

也就是说，从“能不能做出来”这个角度，答案是：

> 能，但成本会高很多，开发效率会低很多，而且不适合这个项目当前的目标。

### 为什么当前项目不选 C 语言

#### 1. 这个项目的核心不是底层算力，而是业务整合

这个系统重点在于：

- Web 登录注册
- 数据库模型
- 模板渲染
- 管理后台
- 实时消息
- 与 LiveKit 的服务端集成

这些需求更偏 Web 应用开发，而不是底层系统编程。  
Python 在这类场景开发更快，改动更直接。

#### 2. C 语言缺少成熟的高层 Web 开发体验

如果用 C 语言，你要自己处理很多原本 Flask 已经帮你做好的事情，例如：

- 路由组织
- 会话管理
- 模板系统
- ORM 或数据库访问封装
- 表单与请求解析

能做，但工程复杂度会明显上升。

#### 3. 开发效率和维护成本不划算

课程项目和迭代型项目通常要频繁改需求。  
Python 改后端逻辑、改页面数据、改接口都很快；如果改成 C，很多普通业务改动会变得更重。

#### 4. 安全和稳定实现成本更高

C 语言虽然性能高，但也更容易引入：

- 内存越界
- 悬空指针
- 资源泄漏
- 字符串处理错误

对于一个以业务功能为主的会议系统来说，这些成本并不值得。

#### 5. 音视频重活已经交给 LiveKit

这个项目最复杂、最讲性能的媒体分发层，已经由 LiveKit 这类专业系统承担了。  
也就是说，我们没有必要为了“追求底层性能”把业务后端也换成 C。

### 一句适合答辩现场的总结

你可以直接这样说：

> C 语言当然可以做服务器和音视频系统，但这个项目的重点是 Web 业务、状态同步和会议功能整合，不是自己从零实现底层媒体引擎。  
> 目前用 Python 的 Flask 来做业务层，配合 LiveKit 做媒体层，是开发效率、维护成本和系统复杂度之间更合理的平衡。

## 9. 如果老师继续追问：那 C 语言适合做什么

你可以继续补一句：

> 如果我要自己做底层媒体服务器、编解码模块、RTSP/WebRTC 网关、极致性能的网络组件，那 C 或 C++ 会更合适。  
> 但当前项目主要是上层会议业务系统，所以选 Python 更符合项目目标。

## 10. 收尾总结

最后可以这样结束：

> 这个项目不是只做了一个视频播放页面，而是把用户系统、房间系统、实时同步、多人音视频、聊天附件和后台管理整合成了一个完整的在线视频会议应用。  
> 技术上，我把业务层放在 Flask，把实时状态同步放在 Socket.IO，把音视频媒体交给 LiveKit，这样既能保证开发效率，也能保证多人会议场景下的可维护性和稳定性。
