# 答辩讲解文档 / Defense Notes

这份文档按“项目是什么、为什么这样设计、代码如何分工、还有哪些局限”来组织，适合课程答辩时直接展开。中文是主讲版本，英文部分可用于 GitHub 展示或补充说明。

This document is organized around what the project is, why it was designed this way, how the code is split, and what limitations still remain. Chinese is the primary presentation language, while English is included for GitHub readability and backup explanation.

## 1. 一句话介绍 / One-Sentence Introduction

中文：

这是一个基于 `Flask + Flask-SocketIO + LiveKit` 的在线视频会议系统，整合了账号体系、房间管理、多人音视频、屏幕共享、聊天附件、管理员后台和会议记录等功能。

English:

This is an online meeting system built with `Flask + Flask-SocketIO + LiveKit`, integrating user accounts, room management, multi-party audio/video, screen sharing, chat attachments, an admin console, and meeting history.

## 2. 项目解决了什么问题 / What Problem It Solves

中文：

这个项目不是单纯地“把视频播出来”，而是把一个会议系统常见的上层需求串成了完整流程：

- 用户注册、登录、登出
- 创建会议和加入会议
- 房间内音视频互动
- 屏幕共享与聊天附件
- 主持人与管理员控制
- 会议历史与基础运维能力

English:

The project is not just about “showing video on a page.” It combines the upper-layer capabilities a real meeting system usually needs:

- user registration and login
- meeting creation and join flow
- in-room audio/video interaction
- screen sharing and chat attachments
- host and admin controls
- meeting history and basic operational support

## 3. 为什么选这套技术栈 / Why This Stack Was Chosen

### Flask

- 中文：负责登录注册、数据库模型、页面路由、后台管理、文件上传、LiveKit token 下发等业务逻辑。
- English: Flask handles business logic such as auth, database models, page routes, admin management, file upload, and LiveKit token issuance.

### Socket.IO

- 中文：负责成员加入/离开、聊天广播、主持人动作、共享焦点与界面状态同步。
- English: Socket.IO handles member join/leave events, chat broadcasting, host actions, share focus, and UI state synchronization.

### LiveKit

- 中文：负责真正的音视频媒体传输，包括摄像头、麦克风、屏幕共享和远端轨道订阅。
- English: LiveKit handles the actual media transport, including camera, microphone, screen sharing, and remote track subscription.

## 4. 为什么不用浏览器 Mesh / Why Not Browser Mesh

中文：

如果每个用户都和所有其他用户直接建链，随着人数增加，连接数量、带宽压力和浏览器负载都会迅速上升。当前项目改用 LiveKit 的 SFU 架构后，每个用户只需要把自己的媒体发到服务器一次，再由服务器转发给其他参与者，更适合多人会议。

English:

In a browser mesh design, each participant connects directly to every other participant. As room size grows, connection count, bandwidth pressure, and browser load all increase quickly. With LiveKit SFU, each participant publishes media once to the server, which is better suited for multi-party meetings.

## 5. 代码结构怎么讲 / How to Explain the Code Structure

### 后端 / Backend

- `app.py`: 当前核心后端文件，包含 Flask 初始化、SQLAlchemy 模型、页面路由、REST API、Socket.IO 事件和运行时配置。

### 模板 / Templates

- `templates/`: 登录、注册、首页、房间页、管理后台等页面模板。
- 房间页重点拆分为 `_room_layout.html` 和 `_room_scripts.html`，分别负责结构和脚本初始化。

### 前端脚本 / Frontend Scripts

- `static/room_livekit.js`: LiveKit 房间连接与媒体控制
- `static/room_ui.js`: 房间卡片布局与共享焦点
- `static/room_chat.js`: 聊天与附件渲染
- `static/room_diagnostics.js`: RTC/LiveKit 诊断摘要
- `static/room_utils.js`: 通用工具函数

### 国际化 / Internationalization

- `translations.py`: 中英文翻译表
- `check_i18n.py`: 模板文案检查脚本

## 6. 用户完整使用流程 / End-to-End User Flow

### 第一步：登录 / Step 1: Login

中文：用户输入账号密码，后端验证后建立登录会话，并通过 `session_version` 管理同账号会话有效性。

English: The user logs in, the backend validates the credentials, and a session is created. `session_version` helps control account-session validity.

### 第二步：创建或加入会议 / Step 2: Create or Join a Meeting

中文：用户可以创建会议，也可以输入房间号和密码加入已有会议。数据库记录会议信息，运行时内存记录在线成员状态。

English: Users can create a meeting or join an existing one with room credentials. The database stores meeting data, while runtime memory stores online participant state.

### 第三步：进入房间 / Step 3: Enter the Room

中文：进入房间后，前端先通过 Socket.IO 声明自己进入房间，再向后端申请 LiveKit token，并连接到 LiveKit 房间。

English: After entering the room page, the frontend first joins through Socket.IO, then requests a LiveKit token from the backend and connects to the LiveKit room.

### 第四步：会议交互 / Step 4: Meeting Interaction

中文：用户可以开关摄像头和麦克风、共享屏幕、发送聊天和附件、使用虚拟背景、进行浏览器录屏。主持人和管理员还拥有额外控制能力。

English: Users can toggle camera and microphone, share the screen, send chat messages and attachments, use virtual background, and record the meeting locally. Hosts and admins have extra control capabilities.

## 7. 项目亮点 / Project Highlights

### 1. 分层清晰 / Layered Design

- 中文：业务层、状态层、媒体层职责已经拆开，不再把所有逻辑都塞进一套浏览器端 WebRTC 代码里。
- English: Business logic, state sync, and media transport are separated instead of being pushed into one browser-side WebRTC flow.

### 2. 功能完整 / More Than a Demo

- 中文：项目不仅能开视频，还包含账号体系、历史记录、附件权限、管理员后台等完整功能。
- English: The project is more than a video demo; it includes accounts, history, attachment permissions, and an admin console.

### 3. 有诊断与运维意识 / Diagnostics and Operations Awareness

- 中文：房间里有 RTC 诊断面板，后端有管理员统计、密码找回申请处理、录屏转封装接口，说明设计目标不仅是“能跑”，也考虑了维护。
- English: The project includes an RTC diagnostics panel, admin stats, password reset handling, and recording remux support, which shows attention to maintainability rather than only basic functionality.

## 8. 当前局限 / Current Limitations

### 1. 在线态仍在内存中 / Presence Still Lives in Memory

- 中文：服务重启会丢失在线态，多实例扩展不安全，后续更适合迁移到 Redis。
- English: Online presence is lost on restart and is not safe for multi-instance scaling. Redis would be a better future target.

### 2. `app.py` 仍然偏大 / `app.py` Is Still Large

- 中文：当前适合课程项目开发节奏，但随着功能增加，最好按认证、房间、聊天、后台等域继续拆分。
- English: This is acceptable for a course project, but future maintenance would benefit from splitting logic by auth, room, chat, admin, and related domains.

### 3. 高资源功能会互相影响 / Heavy Features Can Interfere With Each Other

- 中文：虚拟背景、屏幕共享、录屏都会抢占浏览器或服务端资源，弱设备上要优先保证稳定性。
- English: Virtual background, screen sharing, and recording all compete for browser or server resources, so stability should be prioritized on weaker devices.

## 9. 如果老师问：为什么不用 C 语言 / If Asked: Why Not C

### 直接结论 / Direct Answer

中文：

可以用 C 语言做一部分底层组件，但不适合把当前这个项目整体改写成 C。这个项目的核心挑战是 Web 业务整合，而不是自己从零实现底层媒体引擎。

English:

Some low-level components could be written in C, but rewriting this project in C would not be appropriate. The core challenge here is web application integration, not building a media engine from scratch.

### 具体理由 / Reasons

1. 这个项目重点是账号、房间、权限、数据库、模板和后台，不是纯底层性能竞赛。  
   The project focuses on accounts, rooms, permissions, database work, templates, and admin flows rather than raw low-level performance.

2. Flask 这类框架能显著提高 Web 业务开发效率，而 C 语言在路由、会话、模板、ORM 这类层面工程成本更高。  
   Frameworks like Flask dramatically improve web development speed, while C would make routing, sessions, templating, and database abstraction much heavier.

3. 音视频重活已经交给 LiveKit，没必要为了“性能”把业务后端也换成 C。  
   The heaviest media work is already delegated to LiveKit, so moving the business backend to C would not bring proportional value.

4. C 的内存管理和安全实现成本更高，不符合当前课程项目和快速迭代场景。  
   C carries higher memory-management and safety costs, which is a poor fit for a course project and rapid iteration.

### 适合现场回答的一句话 / A Short Oral Answer

中文：

这个项目的重点是把会议业务系统做完整，而不是自己从零写媒体服务器；所以我用 Python 负责业务层，用 LiveKit 负责媒体层，这是开发效率、维护成本和系统复杂度之间更合理的平衡。

English:

This project is about building a complete meeting application rather than writing a media server from scratch, so using Python for the business layer and LiveKit for the media layer is the more practical balance between development speed, maintenance cost, and system complexity.

## 10. 收尾总结 / Closing Summary

中文：

这个项目把用户系统、房间系统、实时状态同步、多人音视频、聊天附件和后台管理整合成了一个完整的在线视频会议应用。技术上，我把业务放在 Flask，把实时状态同步放在 Socket.IO，把音视频媒体交给 LiveKit，这样既能保证开发效率，也能提高多人会议场景下的可维护性。

English:

This project combines user management, rooms, real-time state synchronization, multi-party audio/video, chat attachments, and administration into a complete online meeting application. Technically, Flask handles business logic, Socket.IO handles real-time state, and LiveKit handles media transport, which gives a practical balance of development speed and maintainability.
